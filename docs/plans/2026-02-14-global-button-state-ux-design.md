# グローバルボタン状態UX改善設計（2026-02-14）

## 目的
- 無効ボタンと有効ボタンの視覚差を明確にする。
- ホバー時・押下時のフィードバックを全体で統一し、操作感を改善する。
- 既存の機能ロジックを崩さず、段階導入で安全に適用する。

## スコープ
- 対象:
  - 問題作成ウィザード
  - ロビー主要操作ボタン
  - 設定画面の主要操作ボタン
- 非対象:
  - ゲーム中の選択肢ボタン等、特殊UI（別途検討）
  - 入力フォーム構造の全面改修

## 方針
- 段階導入（推奨）で進める。
- `QMAClone.css` にボタン状態トークンを追加し、既存クラスへ順次適用する。
- UiBinder 側の変更は最小限にし、既存クラス名を可能な限り維持する。

## コンポーネント設計
### 1. トークン層
- ボタンの背景色、文字色、枠線色、影、遷移時間を定義する。
- 状態別トークン:
  - `default`
  - `hover`
  - `active`
  - `disabled`

### 2. ベース層
- 共通ルール:
  - 角丸、余白、最小高さ
  - トランジション
  - フォーカスリング（`focus-visible`）

### 3. バリアント層
- Primary / Secondary（必要なら Danger）
- 既存の `.creationButtonPrimary`, `.creationButtonSecondary` を起点に展開

## 状態仕様
- `:hover`:
  - 背景、枠線、影を同時変化させる。
- `:active`:
  - `transform: translateY(1px)` と内側影で押下感を出す。
- `:disabled`:
  - 専用配色（背景/文字/枠）を適用し、`opacity` 依存を避ける。
  - `cursor: not-allowed` を適用する。
- `:focus-visible`:
  - キーボード操作時に明確なフォーカスリングを表示する。

## 実装順序
1. CSS トークンと共通状態スタイルを追加
2. 問題作成ウィザードへ適用
3. ロビー主要操作へ適用
4. 設定画面主要操作へ適用

## データフロー
- 既存の `enabled/disabled` 判定ロジックを真実源として維持する。
- Java 側の活性/非活性制御を変更せず、CSS は `disabled` 属性に追従する。

## エラーハンドリング
- 追加変更では例外処理フローは変更しない。
- UI表現の改善のみとし、通信・検証ロジックは現状維持する。

## 検証方針
### 自動検証
- 既存の問題作成関連テストを実行する。
- `gwt:compile` を実行し、スタイル適用後のビルド整合を確認する。

### 手動確認
1. 無効ボタンが有効ボタンと明確に見分けられる。
2. ホバー時に状態変化が視認できる。
3. 押下時に押下フィードバックが視認できる。
4. 問題作成・ロビー・設定の主要ボタンで統一感がある。

## リスクと対策
- リスク: 既存画面で意図しない見た目変化が出る。
- 対策:
  - 段階導入で画面ごとに検証する。
  - 1段階1コミットで戻しやすくする。

## 完了条件
- 対象3画面で `default/hover/active/disabled` が識別可能である。
- 既存機能の操作性に回帰がない。
- build / test / gwt:compile が成功する。
