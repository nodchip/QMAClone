# 非GWTサーバーテスト JUnit5移行 次フェーズ設計

## ステータス

- 補足: 再開時は対象テストの再分類から新規計画として開始する。

- 日付: 2026-02-11
- 対象: `src/test/java/tv/dyndns/kishibe/qmaclone/server` 配下の `@Rule + GuiceBerryRule` 依存テスト
- 方針: Vintage/Jupiter 共存を前提に、移行可能テストのみ段階的に Jupiter 化する

## 目的

非GWTサーバーテストの JUnit5 移行を、実行安定性を維持しながら前進させる。全面移行は目標にしない。`GuiceBerry` 依存が強いテストは Vintage 維持を正式運用とし、Jupiter 化できるテストを継続的に増やす。

## 分類アーキテクチャ

テストを次の3区分で管理する。

1. `移行可 (Jupiter)`
- `@Inject` 中心で GuiceBerry 固有ライフサイクル依存が薄い
- 外部依存（DB/native/ネットワーク）が小さい

2. `保留 (Vintage)`
- 既存挙動を維持すれば安定実行できる
- 直ちに Jupiter 化する合理性が低い

3. `要Extension`
- GuiceBerry が担う前後処理、注入、リソース管理を Extension で再現する必要がある

分類は以下4軸で機械的に判断する。
- 注入方式: `@Inject` のみで足りるか
- 外部依存: DB / native (`zinnia.dll`) / ネットワーク
- 実行安定性: Mockito strictness、乱数、時刻依存
- 保守性: 実行時間、失敗時の切り分け難度

## 実行戦略

実行経路を分離する。

- 第1層: Jupiter 対象
- 第2層: Vintage 維持対象（保留 + 要Extension）

`要Extension` は実装完了まで Vintage 固定とする。保留は先送りではなく運用判断として明示し、Jupiter 移行トリガー（例: Extension 完了、native 非依存化）を定義する。

## 検証フロー

検証は直列で実施する。

1. `compile`
2. `test-compile`
3. Jupiter 対象テスト群
4. Vintage 対象テスト群

完了報告には各段階のコマンドと結果（成功/失敗理由）を記録する。

## 今回合意した次フェーズ実装方針（2026-02-11）

### 1. アーキテクチャ/コンポーネント

次フェーズは「移行可テストを1バッチ10件で増やす」ことを主目標とする。対象は「外部依存ありでも Mockito 置換でローカル完結化できるテスト」まで含める。構成は次の4要素に分割する。

- `(A) 対象選定`: 候補テストごとに `依存タイプ(DB/native/network/file)`、`置換方針`、`想定リスク` を1行で整理する
- `(B) 共通基盤`: 既存の Jupiter + Guice 注入 Extension を固定し、個別テストはこの基盤に合わせる
- `(C) 個別移行`: `@Rule` 依存を除去し、`@ExtendWith` + `@Mock/@InjectMocks` へ統一する
- `(D) 判定記録`: 失敗を `選定ミス` と `実装修正` に分類し、次バッチの候補精度へ反映する

### 2. データフロー/実行フロー

実行は必ず直列で行う。まず候補一覧を作り、各テストに追跡用ID（例: `J5-B10-01`）を付与する。次に10件を選定し、各件で `置換前確認 -> 移行編集 -> 単体実行` を完了させてから次へ進む。失敗時はその場で分類し、`選定ミス` は当該件をバッチ外へ退避、`実装修正` は即修正して再実行する。10件処理後はバッチ全体で再検証し、`テスト名 / 結果 / 失敗分類 / 次アクション` を1表に集約する。

### 3. エラーハンドリング/テスト戦略

失敗は3系統で扱う。

- `環境エラー`: JDK/Maven/依存解決など。テスト品質評価に混ぜず環境復旧を優先
- `選定不適合`: Mockito 置換不能な強依存が残る。今回バッチから外し代替候補を補充
- `実装修正`: 注釈移行漏れ、strict stubbing、assert不整合。修正して同一テスト再実行

検証コマンドは次を固定する。

1. `mvn -q -DskipTests test-compile`
2. `mvn -q "-Dsurefire.skip=false" "-Dtest=<10件カンマ区切り>" "-DfailIfNoTests=false" test`

品質ゲートは「10件すべて成功」とする。未達時は失敗分類に応じて件を入れ替え、再度10件で判定する。
