# client テスト移行バックログ分類 Design

## 背景

`src/test/java/tv/dyndns/kishibe/qmaclone/client` 配下のテストを再分類した結果、次の内訳となった。

- 移行済み: 52
- GWT依存: 17
- 特殊互換（Rule 等）: 1（`ValidatorStressTest`）

通常の JUnit4 -> Jupiter 一括置換で前進できる領域はほぼ消化済みであり、今後は GWT依存と特殊互換の管理が中心になる。

## 方針

### 採用方針

- テストを `移行済み / GWT依存 / 特殊互換` の3状態で管理する。
- 作業ごとに `build -> 対象test -> 状態更新` を直列で実行する。
- GWT依存は無理に移行せず、除外理由と再着手条件を先に固定する。

### 不採用方針

- 都度の場当たり移行（分類を持たない進め方）
- GWT依存を個別に都度検証して失敗を積み上げる進め方

## データフロー

1. バックログから対象を選定（ファイル単位またはディレクトリ単位）
2. 変更または除外判断を実施
3. `mvn -DskipTests package` 実行
4. 対象テスト明示実行（`-Dtest=...`）
5. 結果を設計/計画ドキュメントへ記録
6. 状態を更新してコミット

## ディレクトリ別サマリ（主要）

- `creation/validater`: 移行済み 14、GWT依存 1、特殊互換 1
- `game/input`: GWT依存 2
- `game/sentence`: GWT依存 2
- `creation`: GWT依存 2、移行済み 1
- `packet`: GWT依存 1、移行済み 2
- `setting/theme`、`ranking`、`game/judge`、`game/shuffler`: すべて移行済み

## 優先度（バランス型）

1. `ValidatorStressTest` を特殊互換として正式ラベル化
2. GWT依存17件の除外管理ドキュメント作成
3. 低リスクな GWT依存から段階的に再着手候補を選定

## 完了条件

- 分類表に全テストが登録されている
- GWT依存/特殊互換に除外理由と再着手条件がある
- 実行テンプレート（build -> test -> 状態更新）が運用されている

## 影響範囲

- 本番コードへの影響なし
- バックログ管理ドキュメントのみ更新
