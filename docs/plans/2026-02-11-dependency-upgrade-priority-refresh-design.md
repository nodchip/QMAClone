# 依存関係更新タスク再棚卸し・優先実行 設計

- 日付: 2026-02-11
- 対象: 本番依存 / テスト依存 / ビルド・プラグイン依存
- 方針: `セキュリティ -> 互換性 -> 更新効果` の順で優先度を付与し、上位3件を直列実装する

## 目的

古い依存関係を再棚卸しし、優先度の高い更新から安全に実施する。  
今回はメジャー更新を許容し、必要な追従修正も同タスク内で実施する。

## 1. 全体アーキテクチャ

作業は調査フェーズと実装フェーズに分離する。

1. 調査フェーズ
- `pom.xml` と依存ツリーを基に候補を抽出
- 候補を「本番 / テスト / ビルド・プラグイン」に分類
- 各候補に現行版・目標版・差分種別（メジャー/マイナー/パッチ）を付与

2. 優先度付与フェーズ
- `Security -> Compatibility -> Effect` の固定順で評価
- 同点の場合のみ更新効果で並び替える
- 上位3件を実装対象として確定

3. 実装フェーズ
- 1件ずつ `更新 -> build -> test -> 記録` を直列で実行
- 未解決失敗がある状態で次件へ進まない

## 2. コンポーネント設計

### Dependency Inventory
- 入力: `pom.xml`, `mvn dependency:tree`, plugin定義
- 出力: 更新候補一覧（直接依存/間接依存/plugin依存）

### Priority Evaluator
- 評価軸:
1. セキュリティ（脆弱性・既知リスク）
2. 互換性（Java25/JUnit5移行阻害、API破壊）
3. 更新効果（保守性、将来更新コスト）

### Execution Gate
- 条件: 直前タスクの `build -> test` 成功
- 役割: 失敗時に停止し、原因の切り分けを保証

### Result Ledger
- 記録項目:
1. 更新対象
2. 変更ファイル
3. 実行コマンド
4. 成否
5. 失敗分類
6. 次アクション

## 3. データフロー

`候補抽出 -> 優先度付与 -> 上位3件確定 -> 1件目実装 -> 2件目実装 -> 3件目実装 -> 連結検証 -> 完了記録`

すべて直列で実行し、同時更新は行わない。

## 4. エラーハンドリング

失敗は次の3分類で扱う。

1. 環境要因
- 例: JDK/ローカルライブラリ/実行パス不整合
- 対応: 環境修正を先行

2. 依存互換性
- 例: API差分・挙動差分
- 対応: 当該更新内で追従修正を完了

3. 実装修正不足
- 例: テスト不足・修正漏れ
- 対応: 追加修正後に同コマンドを再実行

## 5. テスト戦略

更新1件ごとに次を実行する。

1. 最小検証（影響範囲の狭い build/test）
2. 回帰検証（既存安定バッチ）

上位3件完了後に連結検証を1回実行する。

## 完了条件

1. 優先度上位3件の更新が完了している
2. 各件で `build -> test` 成功ログがある
3. 実行コマンドと結果が `docs/plans` に記録されている
