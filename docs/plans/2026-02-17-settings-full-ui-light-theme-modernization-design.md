# 設定画面全体UI/UXモダン化（全画面ライトテーマ統一）設計

## 1. 背景と目的
- 設定画面を起点に、左メニューを含む全画面のUI/UXをモダン化する。
- 既存の「濃紺テーマとライトテーマが混在する状態」を解消し、ライトテーマを既定とした一貫した体験に統一する。
- 既存機能（保存・検索・更新・ポップアップ・チャット等）を維持したまま、視認性・可読性・操作一貫性を向上させる。
- 将来のダークテーマ追加を見据え、テーマトークン先行で実装する。

## 2. スコープ
### 2.1 対象
- 設定画面（左メニュー含む）
- ロビー画面
- 検索画面
- 統計画面
- 問題作成画面
- 登録問題一覧
- 類似問題一覧
- チャット
- トースト/エラー通知
- 共通ポップアップ

### 2.2 非対象
- 業務ロジックの仕様変更
- API仕様変更
- テーマ切替UIの追加（今回はライト既定のみ）

## 3. 全体方針
### 3.1 テーマ戦略
- ライトテーマを既定として全面適用する。
- ダークテーマは将来拡張できる構造のみ先行整備し、今回の実装では公開しない。
- 色の直書きは段階的に排除し、用途ベースのテーマトークン参照へ移行する。

### 3.2 実装方式
- 既存CSSへ都度上書きする方式は採用しない。
- テーマトークンを先に定義し、共通部品から置換する。
- 全画面同時改修だが、作業順は「共通部品 -> 画面固有」とする。

## 4. デザイントークン設計
### 4.1 定義カテゴリ
- `color`: 背景、サーフェス、境界線、本文、補助文、アクセント、成功、警告、エラー
- `spacing`: 4/8/12/16/24/32ベースの余白スケール
- `radius`: 入力欄、カード、ポップアップ、トースト
- `shadow`: カード、ポップアップ、浮遊要素
- `typography`: 見出し、本文、補助文、キャプション、数値強調
- `state`: hover/focus/active/disabled/error

### 4.2 命名原則
- 用途ベース命名を採用し、色名ベース命名は避ける。
- 例: `--color-surface-1`, `--color-text-primary`, `--color-accent-primary`, `--color-danger-bg`。

## 5. 共通コンポーネントUIルール
### 5.1 フォーム
- ラベルは原則右寄せ。
- ラベルと入力欄の間隔を固定。
- ラベルと入力欄は上下中央揃え。
- 入力欄（text/select/checkbox）の高さ、境界線、角丸、focusリングを統一。
- 狭幅でも改行崩れしないよう、チェックボックスラベルの改行位置を制御。

### 5.2 ボタン
- 役割を `Primary / Secondary / Danger` の3系統に統一。
- すべてのボタンで `default / hover / active / disabled` を定義。
- 画面間で浮いた配色（白系デフォルト等）を廃止する。

### 5.3 テーブル
- カード内テーブルとして統一。
- ヘッダー、罫線、行hover、選択行、ゼロ件表示を統一。
- 過度な背景ベタ塗りを避け、意味付けはバッジ/文字強調へ寄せる。
- 列ごとのフォントサイズ/太さの不一致を解消する。

### 5.4 通知・トースト・エラー表示
- 表示位置を右下へ統一。
- 成功/警告/失敗の配色ルールを統一。
- 閉じるボタンの配色・hover状態を通知種別と整合させる。
- 「左端アクセント二重」など視覚ノイズは廃止し、枠線ルールを統一する。

### 5.5 ポップアップ
- ヘッダー/本文/フッターの3層構造を統一。
- 空状態/ゼロ件状態の文言を重複なく明確化。
- 次アクションが分かる文言に統一する。

## 6. 画面別適用方針
### 6.1 設定画面
- 左メニューの選択状態を明確化（色、背景、境界線）。
- 本文は「説明」「操作」「結果」を分離し、情報混在を解消。
- 即時保存の導線を維持し、保存中/成功/失敗を同一UIルールで表示。

### 6.2 ロビー・検索・統計・登録問題一覧
- トークン適用により配色と余白を再統一。
- 検索条件フォームのラベル・入力開始位置を厳密に揃える。
- 統計系は数値の意味を保ちつつ、背景依存の強調を抑え、可読性優先へ再設計。

### 6.3 チャット
- ヘッダー、入力部、履歴部のレイアウト基準を統一。
- 狭幅時の見切れ/余白残り/重なりを防止。
- 履歴スクロール、日付改行、404アイコン代替表示など既知課題を吸収。

## 7. 文言設計方針
- 同義の重複表示を排除する。
- ユーザー行動を促す文言へ統一する。
- 画面ごとの命名ゆれ（例: 「参照」「選択」「確認」）を機能単位で統一する。

## 8. 実装手順
1. ライトテーマトークン定義
2. 共通コンポーネント置換（フォーム/ボタン/テーブル/通知/ポップアップ/左メニュー）
3. 画面別調整（設定 -> ロビー -> 検索 -> 統計 -> 作成/一覧 -> 対戦系）
4. 空状態・エラー状態・ゼロ件状態の文言/見た目統一
5. 解像度別検証・回帰検証

## 9. 受け入れ基準（完了条件）
- 全主要画面で色・余白・フォーム・ボタン・テーブル・通知が統一されている。
- 空状態/エラー状態/ゼロ件状態の文言と見た目が統一されている。
- 既存機能に回帰がない。
- 主要解像度（PC標準・狭幅・タブレット）でレイアウト崩れがない。

## 10. リスクと対策
- リスク: 旧セレクタ競合による色戻り。
  - 対策: 旧直書き色の検索と段階置換、差分レビューの徹底。
- リスク: GWT画面固有DOM差による揃え崩れ。
  - 対策: 共通クラス付与を優先し、画面個別補正は最小限に限定。
- リスク: 狭幅時の重なり・見切れ。
  - 対策: `min-width`/`overflow`/`flex-wrap` を明示し、解像度別検証を必須化。

## 11. 検証計画
- 画面単位で以下を確認:
  - フォーム整列
  - ボタン状態遷移
  - テーブル視認性
  - 通知位置と配色
  - 空状態/エラー状態
- 主要解像度でスクリーンショット比較を実施。
- ビルド/テスト/デプロイ検証を順次実行し、結果を記録する。
