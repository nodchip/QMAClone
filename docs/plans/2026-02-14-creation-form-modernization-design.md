# 問題作成フォーム近代化設計（2026-02-14）

## 目的
- Step2〜4 に残る旧デザイン由来の入力フォームUIを統一する。
- Step5 の確認表示も同一トーンで揃え、画面間の連続性を高める。
- 既存の遷移・バリデーション・RPC挙動は維持し、見た目中心で改善する。

## スコープ
- 対象:
  - Step2（基本情報）
  - Step3（問題文）
  - Step4（解答設定）
  - Step5（確認）内の入力風コンポーネント
- 非対象:
  - フォーム機能追加
  - バリデーション仕様変更
  - サーバー側API変更

## 構造方針
- 既存の `Grid / FlexTable` 構造は維持し、外観をカード化して統一する。
- 3層構造で整理する。
  - セクションカード
  - 行グループ
  - 入力コントロール
- 既存クラス (`gridFrame`, `gridNoFrame`, `gridFontNormal`) を直接置換しきらず、
  新規スタイルを重ねて段階的に移行する。

## UI仕様
### 配色
- 青系トーンを維持し、背景コントラストのみ強化する。
- 背景: 薄青
- カード: 白寄り
- 境界線: 青グレー
- 見出し: 濃紺

### 入力状態
- 状態は5種に統一する。
  - default
  - hover
  - focus
  - error
  - disabled
- `focus` はリング表示で明確化する。
- `error` は赤系枠とエラーメッセージで明示する。
- `disabled` は色とカーソルで判別可能にする。

### レイアウト
- ラベル列と入力列の整列を統一する。
- 行間・余白を固定し、情報密度を適正化する。
- 関連項目（選択肢群・解答群など）はサブカードで束ねる。

### Step5確認表示
- 確認カードを「入力結果表示」として扱い、Step2〜4 と同一視覚言語に統一する。
- 「編集へ戻る」導線は現仕様を維持し、カード内の視認性を改善する。

## 実装方針
1. `QMAClone.css` にフォーム共通トークンと基盤クラスを追加
2. `WidgetProblemForm` に最小限のstyle付与を行い、Step2〜4へ適用
3. `CreationUi`（Step5確認）側へ同トークンを適用

## データフローと互換性
- Java 側ロジックは現状維持。
  - ステップ遷移
  - バリデーション
  - RPC
- 見た目のみ変更し、機能回帰を防ぐ。

## 検証方針
### 自動検証
1. `mvn compile`
2. `mvn "-Dtest=CreationUiTest,CreationUiStep4SummaryTest" "-DfailIfNoTests=false" test`
3. 必要時 `mvn "-Dgwt.skipCompilation=false" gwt:compile`

### 手動確認
1. Step2〜5 の遷移中に見た目の統一感がある。
2. 入力欄の hover / focus / disabled / error が識別できる。
3. Step5 の確認表示が編集画面と連続した体験になっている。
4. 既存挙動（入力・遷移・送信）が変わっていない。

## リスクと対策
- リスク: 旧スタイルとの競合で一部崩れる。
- 対策: 新規スタイルは名前空間を `creationForm*` 系に限定し、段階適用で確認する。

## 完了条件
- Step2〜5 の入力系UIが同一トーンで統一される。
- ボタン以外の入力コンポーネントでも状態識別が可能になる。
- 自動検証と手動確認を通過する。
