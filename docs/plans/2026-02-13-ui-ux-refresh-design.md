# UI/UX刷新設計（ロビー先行）

作成日: 2026-02-13  
対象: QMAClone 全体UI/UX（第一段階はロビー）

## 1. 合意済み方針
- 最優先: 見た目の現代化
- デザイン方向: 近未来ゲーム風（深い紺 + シアンアクセント）
- 演出強度: 中程度（操作時中心、常時発光は避ける）
- 変更範囲: 大幅変更可
- 制約: 完成度優先

## 2. 進行アプローチ
推奨アプローチは「情報設計先行 -> UI実装」の二段階とする。  
理由は、見た目先行より手戻りが少なく、完成度優先の要求に合うため。

- Phase A: 情報設計（表示優先順位と配置）
- Phase B: ビジュアル実装（トークン、コンポーネント、動き）
- 適用順: ロビー完成 -> 横展開

## 3. ロビー情報設計
ロビーの主目的は「すぐ対戦開始の意思決定を支援すること」と定義する。

### 3.1 画面構成
- 左: グローバルナビ
- 中央: ロビー本体（主導線）
- 右: 常設チャット

### 3.2 中央パネル構造
- 状態バー: 待機状況、全体対戦待機人数
- メインCTA: `COM対戦` / `全体対戦` / `イベント対戦` / `テーマモード`
- 条件カード: ジャンル・形式・難易度・イベント/テーマ
- 補助情報フッター: レーティング、平均順位、プレイ回数、直近スコア

### 3.3 情報優先順位
1. 状態（今プレイできるか）  
2. 対戦開始CTA（どのモードで始めるか）  
3. 条件（細かい調整）  
4. 補助情報（評価・履歴）

## 4. ビジュアルシステム
### 4.1 デザイントークン
最低限のトークン群を定義する。

- `--bg-*`: 背景
- `--surface-*`: パネル/カード
- `--accent-*`: 強調色（シアン）
- `--text-*`: 文字階層（見出し/本文/補助）
- `--border-*`: 枠線
- `--shadow-*`: 影とグロー

### 4.2 コンポーネント規約
- ボタン種別: `primary` / `secondary` / `ghost`
- CTA4ボタンは同サイズ・同間隔で統一
- 状態バー/条件カード/フッターを部品化
- CSS命名は `lobby-*` 接頭辞で衝突回避

### 4.3 アニメーション規約
- ホバー/アクティブ/フォーカスのみ
- 常時グローは不可
- 時間は 150ms - 220ms に統一

## 5. レイアウト仕様（重要）
中央パネルは縦フレックスで以下を必須とする。

- ヘッダー（状態）: 固定高
- ボディ（CTA + 条件）: 余白吸収
- フッター（レーティング）: 常に下端

これにより「補助情報が下端に固定されない問題」を構造的に防止する。  
右チャットはレイアウト参加型とし、中央と重ならないことを必須条件にする。

## 6. 実装フェーズ
### Phase 1: 基盤
- ロビー専用トークン/CSS追加
- 既存画面への影響を避けるためスコープ限定

### Phase 2: ロビー再構成
- `LobbyUi.ui.xml` のDOMを新構造へ再配置
- 既存機能ロジックは変更しない

### Phase 3: 演出/操作感
- ホバー発光、フォーカス可視化、条件開閉の視認性向上

### Phase 4: 横展開
- 他タブへ共通部品を展開
- 画面ごとに段階適用

## 7. 検証計画
各フェーズで以下を直列実行する。

1. `mvn compile`
2. `mvn test`
3. `mvn "-Dgwt.skipCompilation=false" gwt:compile`
4. デプロイ

UI観点の確認項目:
- 主要解像度で崩れない
- 中央パネルとチャットが重ならない
- フッターが中央パネル下端に固定される
- キーボードフォーカスが視認できる
- CTA到達導線が短い

## 8. リスクと対策
- GWTの既存DOM制約（`TabPanel`/`HorizontalPanel`）  
  対策: DOM実体を前提に段階置換する
- 旧CSS干渉  
  対策: `lobby-*` スコープ化、全体セレクタ禁止
- 実装波及（`LobbyUi.java` への影響）  
  対策: まずレイアウトのみ差し替え、機能ロジックは維持

## 9. 完了条件
- ロビーで新デザインが適用され、主要導線が破綻しない
- build/test/gwt:compile/deploy が成功
- 次画面へ展開可能なトークンと部品規約が確立している
