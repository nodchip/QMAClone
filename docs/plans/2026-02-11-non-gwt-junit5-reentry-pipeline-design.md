# 非GWTサーバーテスト 再投入パイプライン設計

- 日付: 2026-02-11
- 対象: `DatabaseTest` / `FullTextSearchTest` / `RecognizerZinniaTest`
- 方針: 既存10件の安定バッチを守りつつ、低リスクから段階的に再投入する

## 目的

初回10件バッチから除外した3テストを、失敗切り分け可能な順序で再投入する。目的は「本数を増やす」ではなく「再投入可否を明確に判定できる運用」を確立すること。

## 1. アーキテクチャ/コンポーネント

再投入は次の4コンポーネントで構成する。

1. `Baseline Gate`
- 既存10件バッチを基準セットとする。
- 再投入作業中もこの基準が崩れた場合は直ちに停止する。

2. `Re-entry Modules`
- Stage-1: `DatabaseTest`（最小データセット固定）
- Stage-2: `FullTextSearchTest`（一時インデックス毎回生成）
- Stage-3: `RecognizerZinniaTest`（`zinnia.dll` ロード確認 + 最小推論1ケース）

3. `Failure Classifier`
- 失敗は `環境エラー / 選定不適合 / 実装修正` の3分類で管理する。

4. `Rollback Rule`
- 未解決失敗が残る段階で停止し、前段成功状態を維持する。

## 2. データフロー/実行フロー

実行は直列で固定し、段階をまたいで同時実行しない。

1. Baseline 実行（既存10件）
2. Stage-1 `DatabaseTest` 単体 -> 10件+Stage-1 連結
3. Stage-2 `FullTextSearchTest` 単体 -> 10件+Stage-1+Stage-2 連結
4. Stage-3 `RecognizerZinniaTest` DLLロード確認 -> 最小推論 -> 全連結

各段階で `docs/plans` に次を記録する。
- 実行コマンド
- 成否
- 失敗分類
- 継続可否

## 3. エラーハンドリング/テスト戦略

### 失敗分類ルール
- `環境エラー`: ライブラリ不在、権限、パス不整合、JDK差異など。コード変更より環境復旧を優先。
- `選定不適合`: 想定した最小化戦略で再投入不能。対象を保留に戻して次段階へ進まない。
- `実装修正`: テストコードや初期化手順の不足。同一段階で修正・再実行する。

### 品質ゲート
1. 既存10件バッチ成功
2. 段階単体成功
3. 段階連結成功
4. 実行ログ記録完了

4項目すべて満たした場合のみ「段階再投入成功」と判定する。

## 対象外3件の再投入条件

1. `DatabaseTest`
- テスト専用最小データセットで再現可能であること

2. `FullTextSearchTest`
- 一時インデックス生成が実行ごとに完結し、外部状態へ依存しないこと

3. `RecognizerZinniaTest`
- `zinnia.dll` ロード確認成功
- 最小推論1ケース成功

## 運用上の決定事項

- 3件の優先順は `DatabaseTest -> FullTextSearchTest -> RecognizerZinniaTest`
- 差し替えを許可するのは `選定不適合` の場合のみ
- 差し替え時は、差し替え前後のテスト名・理由・再実行コマンドを対象一覧へ必ず追記する
